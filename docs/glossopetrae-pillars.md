## Per-pillar detail

### P1-context — NULL
**Mechanism:** P1 asks whether skinning inbound context into a GLOSSOPETRAE surface compresses what the model reads, lossless-first. The lever would be re-expressing a program or spec or doc in the generated language across the opacity dial and feeding the skinned form as context to spend fewer tokens. That mechanism does not exist. GLOSSOPETRAE's knob is opacity, meaning legibility, not length, meaning token count. The opaque glyph-soup levels where the headline decoupling lives are non-ASCII glyphs, and frontier byte-pair tokenizers fragment each multibyte glyph into several tokens, so the opaque surface inflates token count rather than shrinking it. FINDINGS already flags the tell that Shannon entropy goes the wrong way for glyph-soup. The surface is also a faithful bijection of the original, carrying identical information at identical granularity, so there is no semantic compression either. No lossless length win exists in the concept as built.

**Expected effect:** I tokenized one arithmetic program at every opacity level on two encoders. The readable conlang and the cryptic ASCII level cost about the same number of tokens. The glyph-soup level cost roughly double the conlang. Plain English intent cost the fewest of all, well under half the conlang. So the opaque surface costs about twice the conlang and several times natural language for the same content, because each alien glyph is multibyte and splits into multiple tokens. Opus and Haiku also strip whole categories of exotic glyphs during normalization, so a heavily glyphic inbound surface can be partly deleted before the model reads it, which is lossy. The one exception is the cryptic ASCII level, which is roughly token-neutral, so the conlang idea is not the problem, the opacity dial is. No level beats plain language, and inbound context usually already is plain language.

**Test gate:** Generalize the probe I ran. Take several representative inbound payloads such as a code file, an API spec, and a doc. Skin each at every opacity level using CodeSkin, then tokenize all of them plus the raw original with the target model's tokenizer. The gate for a real lever is that the median skinned token count comes in below the raw token count at some opacity level, while the round trip stays an exact match so it is lossless, and task accuracy on the skinned context stays non-inferior. Use tiktoken with the relevant encodings over the surfaces and assert that some level beats raw. In my run every opaque level failed and only the cryptic ASCII level tied. A pass needs that inequality to flip, which glyph-soup cannot achieve because every multibyte glyph is at least one token and usually more.

**Caveat:** Null specifically for compress inbound context, lossless-first. Two boundaries. First, the concept optimizes the wrong axis for P1, since opacity is not length. A genuine token-compression scheme such as an agreed short-token vocabulary or byte-pair-aware aliasing of frequent multi-token spans down to single cheap tokens is a different mechanism that GLOSSOPETRAE does not provide, so this is not evidence against context compression in general, only against this surface. Second, the token-neutral cryptic ASCII result hints at a narrow new-direction: a tokenizer-aware ASCII-only skin chosen so each keyword maps to exactly one token could cut tokens, but that abandons the decoupling and opacity property that is the actual finding, and still cannot beat the plain natural-language baseline that inbound context usually already is. Hard constraint respected: no covert or stealth framing, pure token economics. Note also that machine usability is execution-graded code, so even the fact the model can still use the surface does not justify paying double the tokens to feed it.

### P2-linguistics — LIMITED
**Mechanism:** P2 asks whether a denser keyed ORDO *surface* (glyphs/morpheme cipher over the readable grammar) can cut command+output tokens FURTHER without losing decode reliability. The glossopetrae key-vs-structure split maps directly onto ORDO: the readable grammar (`sum txt 3bul conc aud:lay no:preamble`) is ORDO's "structure"; a glyph skin (`Σ … ¬preamble`) is a "key" applied on top. The opacity dial is exactly the L0→L3 idea. The crux mechanism: in glossopetrae the key fixes the surface form of ~66% of tokens but 0% of arrangement, and the structure is what the model composes — the surface is a near-free reskin AT THE MACHINE LAYER. The ORDO question is the token-economy corollary of that: does reskinning the surface buy tokens? It does not, for one hard reason — modern BPE (cl100k/o200k) already assigns ~1 token to short readable directives (`sum`=1, `txt`=1), so a glyph can at best TIE and frequently LOSES (a single exotic glyph shatters into 2-3 tokens; ORDO's own law-1 measured runes=3 tokens, uppercase Greek=2 in cl100k). The headroom for "denser surface" is therefore structurally near-zero above the readable grammar, and where it exists it trades reliability (glyph decode 1.95/2 with silent confident-wrong misreads vs readable 2.00/2). The real linguistic headroom is NOT in the surface dial at all — it is one layer down (the grammar: typed slots, determinatives, no-whitespace, which already cut ~32-35%) and one layer up (the OUTPUT contract: format-by-shape TSV/minified-JSON + ponytail verbosity-cut, the ~55-77% lever that produces the 64% end-to-end figure). glossopetrae pillar-2 honest reframe applies as a hard constraint: its machine-usability gains came from EXECUTION-graded code, and its own ORDO-deanchor note found readable WINS on NL-intent decode (2.00 vs 1.95) — so opaque-surface machine gains explicitly do NOT transfer to natural-language commands, which is what ORDO commands are.

**Expected effect:** Denser keyed surface over readable-ORDO: token delta ≈ 0 to NEGATIVE (measured here: glyph-ORDO 13 tok vs readable-ORDO 12 tok on both cl100k and o200k for the canonical command — the glyph skin is one token WORSE), and decode reliability strictly down (1.95 vs 2.00 /2, with the failure mode being silent agreement on a wrong meaning, the worst kind). So the surface dial is null-to-negative on tokens and negative on reliability: there is no free lunch above the readable grammar. Where headroom genuinely exists and is already booked: grammar structure ≈32-35% on input, output contract ≈55-77% on output (64% end-to-end). The one narrow exception the data leaves open is PHRASE/collocation glyphing (always multi-token in BPE: 因=because, 例=for-example), measured at ~1.59 tok saved/use — real but small and it is a vocabulary lever, not a surface-opacity lever, and it caps at ~1-3% of a stream.

**Test gate:** Runnable gate (verified working this session): from `C:\Users\Linus\Documents\GitHub\ordo`, run `PYTHONIOENCODING=utf-8 python -c "import sys;sys.path.insert(0,'tools');from tokcost import cost;..."` over a matched triple {English verbose, readable-ORDO, glyph-ORDO} of the SAME command across the 20-prompt set in `tests/prompts-ordo-readable.txt` vs `tests/prompts-ordo.txt`. GATE: glyph-surface must beat readable-surface by a margin that survives the reliability loss. PASS-condition for "real-lever" = mean(glyph_tokens) < mean(readable_tokens) by >5% AND blind zero-shot decode fidelity (3 fresh agents read `ORDO.md` cold, judge vs hidden originals, /2) for glyph >= readable. Observed: glyph token delta is ~0 to negative (13 vs 12 on the canonical line) and glyph decode is LOWER (1.95 vs 2.00) — the gate FAILS, confirming null. Full reproduction is the existing C2 deanchor harness; the token half is the one-liner above, the decode half is the blind-judge protocol in `docs/BUILD-LOG.md`.

**Caveat:** All token numbers are GPT-tokenizer proxies (tiktoken cl100k/o200k); Claude and Gemini tokenizers are proprietary and could shift the glyph-vs-readable margin — a glyph that shatters in BPE might be 1 token elsewhere, so re-validate per target model before declaring the surface dial dead universally (it is dead on the measured tokenizers). The decode-reliability numbers ran on one frontier model and one ~1.9k-token spec paste (`ORDO.md`); they are not large-N. Critically, do not import glossopetrae's high machine-usability-at-L3 result as license for an opaque ORDO command surface: that result is EXECUTION-graded code, and glossopetrae's own deanchor probe found readable beats opaque on NL-intent decode — ORDO commands are NL-intent, so the opacity gains do not transfer. The pillar is "limited" not "null" only because two adjacent, non-surface levers (grammar structure, output contract) carry real headroom and a third (phrase glyphing, ~1.59 tok/use) is small-but-positive; the specific claim under test — a denser KEYED SURFACE cuts tokens further without losing reliability — is false on the evidence.

### P3-speed — LIMITED
**Mechanism:** Two distinct speed channels exist; both are weak here, for different reasons.

(A) Per-token compute cost. Wall-clock for an LLM is dominated by token COUNT (prefill + decode steps), not by which glyph each token is. The GLOSSOPETRAE opacity dial (L0 readable -> L3 glyph-soup) re-skins the SAME underlying program structure: same statements, same call graph, same token ORDER (FINDINGS §4/S2: "L3 preserves token order, grammar"). So the surface change does not shorten the program. Worse, the only surface-cost number in the corpus cuts against speed: the legibility study found Shannon entropy goes the WRONG way (4.40 -> 3.95 bits/char at L3) "because repeated single glyphs," and the design note in the meta-context (ORDO glyph measurement: runes = 3 tokens worst-case, lowercase Greek = 1 token) shows non-ASCII glyphs frequently tokenize to MORE BPE tokens than the ASCII keyword they replace. A multi-byte glyph that splits into 2-3 tokens inflates both prefill and decode. So the opacity dial is plausibly token-NEGATIVE (slower), and at best token-neutral.

(B) The hard-program effect (SURPRISE) as an indirect speed lever. This is the only real angle. FINDINGS §2b: on HARD programs, L0-hard usability is 20-53% while L3-hard is 97-100%. If forcing the model onto an explicit non-English grammar raises first-pass correctness on hard structured tasks, it cuts RETRIES. Fewer failed attempts = lower end-to-end wall-clock for an agentic loop that re-runs until the program executes. Speed gain here is a second-order consequence of quality (P-quality pillar), not a per-call latency property.

**Expected effect:** Direct per-call latency: null to slightly negative. Same structure + same token order + glyph tokens that often expand under BPE means output token count is flat or higher at L3, so a single generation is no faster and likely marginally slower. The corpus has ZERO wall-clock or token-count data on the in-scope coding arm (PAPER.md line 864 explicitly declines to report wall-clock/cost "because per-vendor rates drift"; the frontier_*.json result files carry only study/model/seeds/legibility/usability/zeroshot/blind/contamControl keys, no token counts), so any direct-speed claim is currently unmeasured AND mechanistically unmotivated.

Indirect throughput via retry reduction: potentially real but modest and entangled. If L3-hard's +44-77pp first-pass usability over L0-hard holds, an execute-until-correct harness does fewer rounds. For a task with first-pass p, expected attempts ~ 1/p; moving p from 0.20 (L0-hard) to 0.97 (L3-hard) drops expected attempts from ~5 to ~1.03, a ~4.8x reduction in generation rounds on the hard slice only. That is a genuine wall-clock win in an agentic loop, but it is a relabeling of the quality result, applies only to hard programs, and assumes each attempt costs roughly equal time.

**Test gate:** Add token/latency instrumentation to the existing in-scope harness (experiments/legibility_frontier.mjs and frontier_full.mjs) — the covert-channel arm already records tokensSent/tokensRecv per trial, so port that block. Two gates:

GATE 1 (per-call cost, the direct claim): For each opacity level L0/L2/L3 over the same seeds/tasks, log completion-token count and provider wall-clock per generation. Compute mean output tokens and ms per level. PASS for a real speed lever only if L3 mean output tokens (or ms) is materially LOWER than L0 at equal usability. Prediction: it will be equal or HIGHER (glyph BPE expansion), i.e. this gate fails -> null/limited.

GATE 2 (retry-throughput, the indirect claim): Run an execute-until-correct loop (cap N=8 attempts) on the HARD task slice at L0 vs L3. Record total attempts and total wall-clock to first passing program per seed. PASS if L3 reaches correct output in significantly fewer attempts AND lower cumulative wall-clock than L0 on hard programs (one-sided test, n>=30 seeds/cell). Prediction: passes on hard, null on easy/med.

Quick sanity check runnable now (no model calls): tokenize the L0 vs L3 skin of the same program with the target model's BPE tokenizer and compare token counts. If L3 token count >= L0, the direct-speed lever is dead on arrival.

**Caveat:** Three honesty bounds. (1) The corpus measures NOTHING about speed — no wall-clock, no token counts on the coding arm (PAPER.md line 864 declines it; result JSONs omit it). Every speed statement here is mechanistic inference plus one adjacent entropy number, not a measured GLOSSOPETRAE result. (2) The one real lever (retry reduction on hard programs) is not a speed finding at all — it is the SURPRISE/quality result wearing a latency hat; if quality is being counted under P-quality, counting it again here is double-counting, and it only applies to the hard slice (L3 = L0 on easy/med, where there are no retries to save). (3) The hard-constraint from the brief bites directly: machine usability is EXECUTION-graded code, and the deanchor result showed opaque surfaces do NOT win on natural-language decode (readable 2.00 vs opaque 1.95). So even the retry-saving effect is confined to structured code-execution tasks and must not be assumed to transfer to NL-command latency. Net: no direct latency lever, a narrow indirect throughput win that is really a quality result, both unmeasured.

### P4-quality — REAL-LEVER
**Mechanism:** P4 is already latent in the existing harness, not hypothetical. `experiments/frontier_full.mjs::usability()` makes the model COMPOSE a program (control flow, recursion, call graph) in the opacity-skinned surface, then execution-grades the unskinned output byte-for-byte. The §2b sweep shows the inversion the pillar claims: on the hard 3-construct task, L0 (readable conlang) scores 20% (Opus) / 53% (GPT-5.5) and L2 27% / 77%, while L3 (glyph-soup) scores 97% / 100%. The readable surface HURTS hard structure; the opaque explicit grammar does not.

The proposed mechanism for WHY: surface opacity forces token-by-token grammar discipline. At L0 the conlang reads like loose pseudo-English, so the model relaxes into English-prior autocomplete — it emits plausible-looking code and trips on the genuinely hard part (operator precedence, recursion base-case, statement order, terminator placement) because the familiar surface lets it pattern-match instead of parse. At L3 every token is an arbitrary glyph with no English affordance, so the model has no prior to coast on: it must hold the spec's explicit token→concept map in working memory and place each symbol deliberately. That deliberateness is exactly what hard structure needs. Opacity acts as a forcing function that converts "write code that looks right" into "emit the precise grammar the key defines."

CRITICAL CONFOUND that decides real-lever vs artifact: the harness holds the natural-language task `desc` constant across opacity, but does NOT isolate whether L0's failure is structural (wrong AST) or surface (right AST, sloppy transcription — a missing terminator, a collided identifier). The faithfulness control (FINDINGS §3.1) shows L0/L1 are where skin round-trips break on adversarial inputs precisely because ASCII tokens collide; L2/L3 are collision-free by construction. So part of L0's 20% could be the model writing CORRECT structure that the readable skin then mis-parses — meaning L0 is penalized for surface ambiguity, not the model's structure being worse. P4 is a real quality lever only if the failure is structural; it is a grading artifact if the failure is transcription.

**Expected effect:** Two outcomes, both informative. (A) REAL-LEVER outcome: when you re-grade by AST-equivalence (structure) instead of exact-output (surface), the L0-hard < L3-hard gap PERSISTS — the readable surface produces genuinely wrong control flow on hard programs (missing recursion base case, wrong loop bound, mis-nested expression), and forcing the model onto the explicit glyph grammar produces structurally correct programs more often. Predicted: structural pass rate stays inverted, e.g. L0-hard ~30-50%, L3-hard ~90%+, gap >35pp. (B) ARTIFACT outcome: AST-equivalence closes the gap (L0-hard structural ~85%+, matching L3) — the model composes the right structure at every opacity, and L0's exec failures were surface collisions the skin couldn't round-trip. Given §3.1 (all faithfulness mismatches are L1-only adversarial, L0 benign programs round-trip clean), I lean toward (A) being partly real but smaller than the raw exec gap: expect the structural gap to shrink from ~70pp to maybe ~25-40pp but not vanish. Magnitude is the whole question — the raw exec number over-states the quality lever because it conflates surface and structure.

**Test gate:** Graft onto the existing harness, do not rebuild it. In `experiments/frontier_full.mjs`, the `usability()` path already produces, per seed: the model's skinned program, the unskinned canonical, and the executed output. Add a STRUCTURAL grade beside the existing exact-output grade: (1) parse both the model's unskinned program and the oracle `src` through `CodeForge`'s parser to ASTs; (2) compare ASTs canonically (normalize identifier names and whitespace, keep shape: node types, child order, operator nesting, call targets). Pass = isomorphic AST even if output differs or a terminator is missing. This separates "wrong structure" from "right structure, surface slip."

Concrete gate (runnable, ~30 seeds/cell, mock first then real):
1. `node experiments/frontier_full.mjs --mock` must still pass (faithful-skin control = oracle is 100% usable+structural everywhere). This proves the AST grader doesn't false-negative on correct programs.
2. Real run: `node experiments/frontier_full.mjs --model anthropic/claude-opus-4.8 --seeds 30`, emitting BOTH grades per cell. Repeat for gpt-5.5.
3. DECISION RULE (pre-registered): compute structural_gap = mean(L3-hard structural) − mean(L0-hard structural). REAL-LEVER if structural_gap ≥ 0.20 AND L0-hard structural < L3-hard structural with non-overlapping ±1.96 binomial CIs at n≥30. NULL/artifact if structural_gap < 0.10 or CIs overlap (the lever was surface-collision grading, not quality). LIMITED if 0.10 ≤ gap < 0.20.
4. Confound scrub: also report exact_gap − structural_gap. If that difference is large, the raw §2b finding over-claims the quality effect by that amount — report it as the honest correction.
Add as a third column in the §2b table; no new files, ~40 lines in the existing sweep.

**Caveat:** Three hard bounds. (1) EXECUTION-GRADED CODE ONLY — the entire effect is measured on programs run by an interpreter to exact stdout. The ORDO de-anchor result (READABLE wins NL-intent decode 2.00 vs 1.95) directly warns that this opaque-helps-hard-structure gain almost certainly does NOT transfer to natural-language commands or any task where "quality" is judged on meaning rather than executed correctness. P4 is a coding-structure lever, not a general quality lever; do not pitch it as "opaque grammar makes the model write better prose/specs." (2) The structure tested is MODEST, TEXTBOOK logic (loop+accumulator, factorial recursion, two-function compose) that is certainly in pretraining (FINDINGS §4, S1 caveat). The claim is "composes familiar structure more reliably under a forcing surface," NOT "composes harder structure." The lever may flatten or reverse on genuinely novel algorithmic structure the model has no prior for — untested and a real risk. (3) The mechanism story (opacity defeats English-prior autocomplete) is an inductive-bias claim, and RELATED_WORK explicitly says NOT to lead with the English-prior story — that hill is contested (Mission:Impossible / GCG) and the data doesn't cleanly support it. So state the EFFECT (opaque surface lifts hard-structure exec pass rate) as the finding and treat the WHY as hypothesis, not result. Finally, n=30 is moderate-confidence (CIs ±7-18pp); a clean read needs the ≥100-seed L0-vs-L3 contrast re-run before publication.

### P5-halluc — LIMITED
**Mechanism:** Two distinct mechanisms, only one of which the bench actually probes.

(A) Code-substrate confident-wrong (testable today, real signal). The grader already separates failure modes: gradeStructured returns {pass, degenerate_precompute, missing_structure, parse_error, wrong_output}. "wrong_output" IS confidently-wrong: the model emitted a complete, parseable program that executes to the wrong answer — a hallucinated solution, not a refusal. The load-bearing datum is the SURPRISE result (FINDINGS §2b, L88-89): on hard programs the readable English-looking surface produces MORE wrong answers than glyph-soup. Opus L0-hard usability 20% vs L3-hard 97%; L2-hard 27% vs L3-hard 97%. Mechanistically, the readable surface (L0 conlang that looks like `zuba s = 0 ; shoa(...)`) sits near the English/real-code manifold, so the model auto-completes from a familiar-but-wrong prior — the "helpful-fix" artifact in grade_rigor.mjs is exactly this: the model abandons the alien spec and writes `function`/`if`/`while` from muscle memory, confidently and wrongly. Glyph-soup (L3) has no nearby prior to fall back on, so the model is forced to deliberately compose against the explicit per-eval grammar it was given, and it gets the STRUCTURE right. The explicit grammar acts as a forcing function that suppresses prior-driven confabulation on hard structure.

(B) NL-intent / factual confident-wrong (the thing "hallucination" usually means): null/untested. The contamination-free framing prevents one narrow failure — the model cannot regurgitate a memorized corpus answer because no corpus exists (identity control 624/624). But that is anti-leakage, not anti-confabulation, and the HARD CONSTRAINT applies: machine usability is execution-graded CODE, and the ORDO deanchor result shows opaque surface does NOT win on NL decode (2.00 readable vs 1.95). There is no abstention/IDK signal anywhere in the harness, so factual hallucination is simply not measured.

**Expected effect:** Real but BOUNDED to structured/code generation. Expect the explicit per-eval grammar to cut confident-wrong specifically where the failure is "the model defaulted to a familiar surface and got the structure wrong" — the measured 70+ percentage-point swing on hard programs (L0-hard 20% -> L3-hard 97% for Opus; 53% -> 100% for GPT-5.5) IS that effect, and it is large, replicated across two model families, and execution-graded (no judge). Reframe it as: forcing generation onto an explicit, unfamiliar grammar reduces prior-driven confabulation on hard composition. Expect NEAR-ZERO transfer to natural-language factual hallucination: the contamination-free property only blocks memorization-leak (a small slice of hallucination), and opaque surface plausibly HURTS NL tasks per the ORDO 2.00-vs-1.95 deanchor. So: real lever for "model writes confidently-wrong structured output," null/unproven for "model states confidently-wrong facts."

**Test gate:** Gate exists in infra; needs one new axis (abstention) to be honest. Concrete runnable gate:

1. EXISTING SIGNAL (run now): from results/frontier_anthropic-claude-opus-4.8.json and gpt-5.5.json, recompute per-cell the fraction of NON-pass that are mode=='wrong_output' (confident-wrong) vs parse_error, using classify() in experiments/lib/grade_rigor.mjs. Hypothesis to confirm: wrong_output RATE is higher at L0-hard/L2-hard than L3-hard. If confident-wrong falls as the grammar gets more explicit/opaque, the lever is real. This is a re-grade of existing runs, zero new model calls.

2. NEW ABSTENTION AXIS (the missing control): add an unsatisfiable/underspecified task variant (spec omits an operator the task needs) and instrument whether the model ABSTAINS ("the spec doesn't define X") vs hallucinates a token. Measure abstention-rate and false-confidence-rate per opacity level, ≥30 seeds/cell, binomial CI. Real lever <=> explicit grammar raises correct-abstention without raising parse_error. Build on frontier_full.mjs grid; reuse gradeStructured's mode taxonomy plus a new mode='abstained'.

3. NL CONTROL (kills the over-claim): run the same opacity dial on a natural-language QA/decode task (not execution-graded) per the ORDO deanchor protocol; if opaque surface does NOT reduce confident-wrong on NL, report the lever as code-only. Decision rule: declare real-lever ONLY for the code axis where wrong_output drops monotonically with explicitness AND abstention rises; declare null for NL if the ORDO direction holds.

**Caveat:** The honest verdict is "limited / real-but-narrow," not a clean win. The strong, replicated effect (explicit grammar cuts confident-wrong STRUCTURE on hard programs, 70+pp) is real but it is CODE/EXECUTION hallucination under an unfamiliar-surface forcing function — it does NOT license the general claim that this reduces factual or NL hallucination. Three hard limits: (1) the HARD CONSTRAINT is binding — machine usability is execution-graded code, and ORDO deanchor (2.00 readable vs 1.95 opaque on NL decode) suggests opaque surface may HURT, not help, NL-intent tasks, so do not extrapolate. (2) Contamination-free only blocks memorization-leak, which is a small and specific slice of hallucination; it does nothing for confabulation from a wrong-but-plausible prior. (3) There is currently NO abstention/IDK signal in the harness, so "confident-wrong" is only measured as wrong_output on tasks that HAVE a right answer — the canonical hallucination test (does it invent an answer when none exists / it doesn't know?) is unbuilt. Until the abstention axis (test #2) is run, the anti-hallucination claim is inferred from the structure-fidelity surprise, not directly demonstrated. New-direction worth noting: the wrong_output-vs-parse_error split is an underused, already-built lens on confident-wrong code generation and is the cleanest thing to publish here.

### P7-arch — LIMITED
**Mechanism:** P7 asks whether composing in an explicit, non-English grammar makes the model reason more structurally about software architecture decisions (rebuild-vs-fix, where-to-cut, module boundaries). There are two candidate mechanisms in the findings, pulling opposite ways.

PRO mechanism (the only real one): the SURPRISE result, FINDINGS.md lines 86-89. On the hard 3-construct task, the READABLE surface HURTS (L0-hard: Opus 20%, GPT-5.5 53%; L2-hard 27%/77%) while glyph-soup does NOT (L3-hard: Opus 97%, GPT-5.5 100%). The proposed reading: a familiar English-shaped surface lets the model pattern-match to memorized snippets and skip explicit structural planning; an opaque grammar denies that shortcut and forces it to build the call graph / control flow from the spec's rules. If that "denied-the-shortcut, plan-explicitly" effect is real and general, then phrasing an architecture problem as an explicit formal grammar (typed module DSL, dependency-graph notation, an ADR schema with hard slots) could push the model from surface analogy ("this looks like a God-object, rewrite it") toward composed structural reasoning ("trace the actual coupling, decide fix-vs-rebuild from the dependency shape").

CON mechanism (the load-bearing constraint, and why this is mostly speculative): the honest reframe, FINDINGS.md lines 20-23 and 143-152 — "structure is the model, surface is a key." The key fixes ~66% of surface tokens and 0% of arrangement; a key-preserving shuffle parses 0/120. The model ALREADY composes correct structure zero-shot at every opacity level; the skin does not add structural reasoning, it just renames tokens. So the default expectation is that opacity is orthogonal to architectural reasoning quality. Worse, the hard constructs that resolved at L3 were "modest, textbook logic" (loop+accumulator, factorial recursion) almost certainly in pretraining (line 150-152). Architecture decisions are the opposite: open-ended, context-laden, no oracle, graded on judgment not on byte-exact stdout.

**Expected effect:** Decompose into the two claims the pillar bundles.

Claim A ("opaque surface itself improves architectural reasoning"): expected effect NULL to slightly negative. The execution-graded result does not transfer. Architecture has no executable oracle — it is exactly the NL-intent-decode regime the hard constraint flags, where the ORDO deanchor found READABLE WINS (2.00 vs 1.95). An opaque architecture description would degrade, not help, because the reader (model or human reviewing the decision) loses the semantic surface that carries domain meaning, and architecture meaning lives in the vocabulary layer (which S2 confirms stays opaque without the key, lines 158-165), not the grammar skeleton. So skinning an architecture brief into glyph-soup is expected to HURT.

Claim B ("forcing composition through an EXPLICIT GRAMMAR — not opacity, just explicitness — improves structural reasoning"): expected effect SMALL-POSITIVE but UNPROVEN, and this is the only defensible read. The SURPRISE result is consistent with "explicit-grammar-beats-familiar-surface ON STRUCTURE-HEAVY TASKS." Mapped to architecture: forcing a rebuild-vs-fix decision through a constrained formal schema (must emit: dependency edges, coupling count, blast-radius set, a typed verdict over {fix, refactor-in-place, strangler, rebuild}, each slot justified from the graph) plausibly suppresses the surface-analogy shortcut the same way the opaque grammar did on hard-L3. The expected lift is the architecture analogue of the +77pp hard-L3 gain — but there is no evidence it is anywhere near that size, because (i) the hard-L3 structure was textbook/pretrained and architecture judgment is not, and (ii) the gain came from DENYING a memorized surface, and a structured ADR schema only weakly denies that. Realistic expectation: a modest, possibly single-digit-percentage improvement in decision consistency and structural justification, mostly attributable to "force explicit intermediate structure" (a known scaffolding/CoT effect) rather than to anything specific to glossopetrae's invented-language finding.

**Test gate:** Runnable gate, three-arm A/B/C on rebuild-vs-fix decisions, execution-anchored where possible to avoid LLM-judge slop.

Corpus: 30-50 real architecture decisions with a defensible ground-truth label. Source them where the right answer is knowable: (a) git histories where a "rebuild" or "strangler-fig" actually happened and post-hoc was right/wrong (commit + outcome), and (b) seeded synthetic codebases with a KNOWN coupling structure (you control the dependency graph, so "fix is cheaper" vs "rebuild is cheaper" has an oracle: e.g. blast-radius > N modules and cyclomatic coupling > K => rebuild). The synthetic arm is the rigorous one because it gives execution-style grading, mirroring FINDINGS §3.4.

Arms, same model, same temperature, n>=30/cell for >=7pp CIs (match the sweep, FINDINGS line 96):
- ARM A (control): free-form English prompt "should we fix or rebuild this? explain."
- ARM B (explicit-grammar, READABLE): require the answer as a filled formal schema — emit dependency edges as `a -> b`, a coupling integer, the blast-radius module set, then a typed verdict token from a fixed enum, each justified by citing the emitted graph. Readable surface, but composition forced through grammar. This isolates Claim B (explicitness) from Claim A (opacity).
- ARM C (opaque-grammar): same schema as B but skinned to L2/L3 glyphs via CodeSkin, with the key provided. This isolates Claim A (does opacity add anything beyond explicitness).

Grade on the synthetic arm by exact match to the oracle verdict (rebuild/fix) AND structural-faithfulness (did the emitted dependency graph match the true one — gradeable deterministically). 

PASS for "real-lever": ARM B beats ARM A on oracle-verdict accuracy by a CI-separated margin (lower bound of the difference > 0 at n=30). 
PASS for "opacity matters": ARM C beats ARM B by a CI-separated margin. 
Predicted result: B > A by a small but real margin (explicit-structure scaffolding works); C <= B (opacity adds nothing or hurts — consistent with ORDO deanchor READABLE-wins on NL). Run cost: ~150 model calls, one afternoon, no human study needed for the synthetic arm.

**Caveat:** The pillar conflates two things and only one is even plausibly real. OPACITY (glossopetrae's actual contribution — the invented surface) is null-to-negative for architecture: the hard constraint is explicit that machine usability is execution-graded CODE, architecture decisions are NL-intent-judgment with no executable oracle, and the ORDO deanchor shows readable surface WINS on NL decode (2.00 vs 1.95). So do not claim the invented-language result improves architectural reasoning — it does not transfer. What survives is weaker and not actually a glossopetrae finding: "forcing the answer through an explicit formal schema improves structural consistency" is the ordinary structured-output / chain-of-thought scaffolding effect, which would work in plain English (ARM B) and gains nothing from opacity (ARM C). The SURPRISE result (L0-hard hurts, L3-hard fine) is the one suggestive thread, but it is bounded by S1's concession that the hard structure was textbook, pretrained logic with a byte-exact oracle — none of which holds for rebuild-vs-fix. The honest verdict is "limited": the test is worth running because the explicit-grammar arm (B) might show a small real lift, but the headline glossopetrae mechanism (opaque per-eval language) is the wrong lever here, and a positive B-result would be evidence for structured prompting generally, not for this paper. Speculative as the brief warned; do not force-fit it into the architecture pillar.

### P8-rework — LIMITED
**Mechanism:** Use CodeForge(seed) to mint a fresh programming language + recursive-descent interpreter per evaluation, render a task in that language's canonical surface (optionally CodeSkin-opacified to L2/L3 glyph-soup), hand the model the per-seed spec, and grade by EXECUTION on the generated interpreter (gradeStructured: exact stdout match AND required control-flow present, rejecting degenerate-precompute). Because the language existed milliseconds earlier and the seed is drawn fresh each run, there is no corpus to memorize and no fixed benchmark string for a model (or a contractor RLHF'ing against your eval) to pattern-match. The identity control (624/624, wrong-seed key never reproduces the answer) is the formal teeth: a "gameable" answer would have to generalize across seeds, which by construction it cannot. The glossopetrae-specific increment over ordinary property-based/fuzz testing is forcing the model OFF the familiar English/Python surface onto a novel surface, so a retrieved solution SHAPE (the memorized arrangement, not just I/O pairs) fails too. Almost all of this is already implemented: experiments/lib/tasks_rigor.mjs builds non-guessable per-seed-literal tasks, experiments/lib/grade_rigor.mjs::gradeStructured does the execution+structure grade, frontier_full.mjs wires the loop. A gate is ~60 lines: for N fresh seeds, build task, prompt model, gradeStructured, fail the run if pass-rate < threshold.

**Expected effect:** Strong and measured where it applies as a CAPABILITY/REGRESSION eval. From FINDINGS §2b at n=30/cell: zero-shot usability on the novel surface is 92-99% (the example is not doing the work), identity control is 100% at all levels, and base-rate guessing on the rigor tasks is provably <5% because the oracle embeds cf.seed. So a model genuinely solving the task passes ~95%+; a model that memorized your old benchmark, or a finetune gamed against a fixed eval set, gets no lift because every run is a language it has never seen. Per-eval generation is the single property in the whole study that transfers cleanly to "un-gameable": it is contamination-free BY CONSTRUCTION, not by scarcity or time-cutoff, which is strictly stronger than EsoLang-Bench (fixed languages) and RELIC (membership not production). The opacity dial is OPTIONAL here and mostly orthogonal to un-gameability: L0-readable is already un-memorizable because the lexicon is per-seed; pushing to L3 buys you the SURPRISE result (glyph-soup does not hurt hard programs, L3-hard 97-100% vs L0-hard 20-53%) which makes the gate MORE stable on hard tasks, but it is not what makes it un-gameable. Net expected effect: a regression gate whose difficulty cannot decay through contamination, with ≤±7pp CIs at 30 seeds on the L3 rows.

**Test gate:** Runnable today against the existing harness. (1) Self-check the task bank is non-guessable: `node experiments/lib/tasks_rigor.mjs --selftest` (asserts oracle == cf.run(src).output per seed and that most-common-output base-rate < 5%). (2) Stand up the gate as a thin wrapper over frontier_full.mjs: for seeds 0..29, cf=new CodeForge(seed); for each rigor task, prompt the model with sk.spec(), grade with gradeStructured(cf, sk, modelProgram, oracle, needs); record pass. Run it TWICE with disjoint seed ranges (e.g. 3000+ and 9000+) on the SAME model — un-gameability predicts the two pass-rates agree within CI (no seed range is "easier" because none was memorized). (3) The discriminating test for the un-gameable CLAIM specifically: take a model deliberately overfit to a FIXED frozen eval (cache 30 prompt→answer pairs, finetune or few-shot them in) and show its pass-rate on FRESH seeds is unchanged vs a clean model, while its pass-rate on the frozen set is inflated. If fresh-seed pass-rate moves under overfitting, the gate is gameable and the lever is null. (4) Sanity gate threshold: fail CI if mean structured-pass < 0.85 at n≥30 (CI ≤±7pp per FINDINGS §2b), which a real capability regression would trip and a contamination artifact would not.

**Caveat:** The honest scope is a CAPABILITY / model-regression eval, NOT a code-correctness CI gate for a product like brain-os. A product's correctness is defined against fixed real specs (the api.py routes, the strand contract); randomizing the language tests the model's coding ability, not whether YOUR code is right, so this never replaces tests/test_*.py. Two further bounds from the findings cut the novelty: (a) machine usability is EXECUTION-graded CODE — the hard constraint says opaque-surface gains do NOT transfer to natural-language intent (ORDO deanchor: readable wins NL decode 2.00 vs 1.95), so a per-eval gate works for code/algorithmic reasoning and is unproven-to-null for "did the model follow this English instruction" evals. (b) Most of the un-gameability is already delivered by ordinary property-based/fuzz testing with randomized inputs and an oracle; the ONLY glossopetrae-specific increment is defeating retrieval of a memorized solution SHAPE by changing the surface grammar, which matters for benchmark-contamination of capability evals but is a narrow add-on, not a new testing paradigm. S2 also weakens it: the grammar skeleton is keyless-crackable (100% of seeds pin ;,=,{,(,) ) so a determined adversary can tell it is imperative code and parse its shape — vocabulary stays opaque, structure leaks. And the headline confidence is 30 seeds/cell (moderate), below the ≥166-seed publication bar, with no human study run. Net: a real, runnable, genuinely contamination-free capability-eval gate — but "limited," because its un-gameable property is narrow (code only, surface-retrieval only) and largely overlaps existing randomized testing rather than founding something new.

### P9-longform — LIMITED
**Mechanism:** see structured fields

**Expected effect:** PILLAR METRIC: Q/T = (fraction of loop runs whose FINAL output passes the byte-exact execution gate) / (total output tokens emitted across ALL passes: drafts + revisions + final). A loop 'wins' iff Q/T_loop > Q/T_singleshot.

Predicted, from the sec2b hard-cell numbers (Opus 4.8, n=30):
- On HARD tasks, an off-English forced-grammar intermediate representation lifts single-pass p from ~0.20 (readable) toward ~0.97, cutting E[passes] from 3.36 to 1.03 (a 3.3x reduction in net tokens) AND raising solved-within-5 from 67% to 100%. Net Q/T improvement is large and driven entirely by fewer wasted passes.
- On EASY/MED tasks the readable surface is already 73-100%, so there is little headroom; expect Q/T parity or a small loss (you pay the glyph token tax for near-zero pass-count benefit). The lever is hard-task-specific.
- The execution gate alone (independent of surface) raises Q/T over any LLM-judge loop by removing judge tokens and judge false-passes; this is unconditional.

So: real lever on the verify HALF (gate) and on hard-task pass-rate (off-English surface); null on the 'opaque surface compresses tokens' half.

**Test gate:** Runnable gate, reusing the existing harness (experiments/frontier_full.mjs + lib interpreter synthesis):

1. Build a loop driver: for each (seed, task) mint a fresh language + interpreter (per-eval), prompt the model to write the program, UNSKIN + EXECUTE, string-compare to oracle. On mismatch, feed back ONLY 'wrong output, expected X got Y' (no spec leak) and re-draft. Cap K=5. Log output-tokens PER PASS.
2. Two arms x 3 difficulties {easy,med,hard}: ARM-R = readable L0 surface, ARM-G = forced-grammar L3 surface. >=30 seeds/cell (match findings n).
3. Metrics per cell: Q = solved-within-5; T = sum output-tokens across passes; report Q/T and E[passes]. PASS CONDITION for the pillar: on the HARD cell, Q/T_G > Q/T_R with non-overlapping bootstrap CIs, AND E[passes]_G < E[passes]_R. NULL if hard-cell Q/T_G <= Q/T_R.
4. Control 1 (compression null): measure tokens/program for L0 vs L3 on identical oracle solutions; confirm L3 >= L0 (glyph tax is real) so any G-win is attributed to pass-count, NOT compression.
5. Control 2 (NL boundary): repeat one cell where the 'gate' is an LLM-judge over natural-language intent instead of execution. Predict the surface lever VANISHES or reverses (ORDO deanchor: readable 2.00 > opaque 1.95). This bounds the claim to execution-graded code.
6. Stat bar: >=30 seeds for directional, flag that publication needs >=166 (findings sec6).

**Caveat:** Three hard bounds. (1) EXECUTION-GRADED CODE ONLY: the entire lever rests on a byte-exact executable oracle and on the hard-program structure result. The findings' own hard constraint plus the ORDO deanchor (readable wins NL-intent decode 2.00 vs 1.95) say the surface benefit likely does NOT transfer to a loop over natural-language deliverables (drafts, plans, copy) — there the loop gets the gate-quality benefit at best and a token PENALTY from any opaque surface. Do not generalize Q/T gains beyond code with a real interpreter. (2) THE COMPRESSION READING IS NULL: glyph-soup costs MORE tokens/program, not fewer; the net-token win is from fewer PASSES (3.3x), never from a denser surface, and it only nets positive while the per-pass glyph tax stays under the computed break-even g*=4.85x (real ~1.5-3x, so it holds, but it is a tax being out-run, not a saving). (3) NARROW + NOISY EVIDENCE: the 20%/97% hard-cell split is n=30, CIs +-7-18pp, one task family (sum/factorial/two-fn compose); 'hard' = 3-construct textbook logic, not real software. The off-English-helps-hard-structure effect is the load-bearing claim and is the least-replicated number in the findings. Also: 'forced grammar' not 'glyph-soup' is what does the work — an explicit non-English IR captures the lever without paying the full glyph token tax, and that decoupling is untested here.

## Bottom line
GLOSSOPETRAE is a **quality + evaluation** insight, not a compression one. The opaque surface INFLATES tokens (P1 null, P2/P3 limited) and does not transfer to NL-intent (re-confirming the deanchor). The two real transfers: **P4** (forcing the model off the familiar surface improves hard-task structure) and **P8** (contamination-free per-eval = un-gameable test gates). Everything else is bounded. The value is captured by the REFEED framework (`spec/framework.md`), not by glyphing the surface.
